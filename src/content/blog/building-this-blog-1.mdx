This is the first post in a series documenting how RayBlog was built. I wanted a personal blog that's fast, easy to write for, and lets me add interactive components without paying a JavaScript tax on every page. Here's the stack I landed on and why.

## Why Astro

I've used React-heavy setups (Next.js, Remix) and fully static setups (Hugo, Eleventy) before. Astro sits at an interesting point between the two.

The core idea is the **islands architecture**: pages are static HTML by default, and JavaScript only ships for the components that explicitly need it. For a blog, that means the post itself is pure HTML — no hydration cost — while interactive pieces like the theme toggle or search bar opt in to JavaScript individually.

The result is a 100/100 Lighthouse score on content pages without doing anything clever. The default is already fast.

### Content-first by design

Astro treats markdown and MDX as first-class citizens. A `.mdx` file in `src/content/blog/` becomes a page automatically. The layout, the reading time, the table of contents — all derived from the file, no configuration glue code needed.

```ts title="src/pages/blog/[...slug].astro"
import { getCollection } from "astro:content";

export async function getStaticPaths() {
  const posts = await getCollection("blog", (p) => !p.data.draft);
  return posts.map((post) => ({
    params: { slug: post.id },
    props: { post },
  }));
}
```

That's the entire dynamic route. Astro calls `getStaticPaths` at build time and pre-renders every post.

## Content Collections

Content Collections are Astro's typed content system. You define a schema with Zod, and every post's frontmatter is validated at build time:

```ts title="src/content.config.ts"
import { defineCollection, z } from "astro:content";
import { glob } from "astro/loaders";

const blog = defineCollection({
  loader: glob({ pattern: "**/*.{md,mdx}", base: "./src/content/blog" }),
  schema: z.object({
    title: z.string(),
    description: z.string().optional(),
    pubDate: z.coerce.date(),
    tags: z.array(z.string()).default([]),
    draft: z.boolean().default(false),
    series: z.string().optional(),
    seriesOrder: z.number().int().optional(),
  }),
});

export const collections = { blog };
```

If a post is missing a required field or has the wrong type, the build fails with a clear error. No silent data problems in production.

## MDX: Components in Prose

Plain markdown gets you a long way. MDX gets you the rest. It lets you import and use components directly in a post:

```mdx
import Callout from "@/components/Callout.astro";

Here's a warning:

<Callout type="warning">This API was deprecated in v3.</Callout>
```

The components render at build time (for Astro components) or hydrate on the client (for React components with `client:*` directives). You get rich interactive content with no special CMS or authoring tool.

## Tailwind CSS v4

Tailwind v4 drops the JavaScript config file. Everything moves into CSS using `@theme` directives:

```css title="src/styles/global.css"
@import "tailwindcss";

@theme inline {
  --color-primary: var(--primary);
  --color-muted: var(--muted);
  --font-sans: var(--font-sans);
}

:root {
  --primary: oklch(0.5 0.26 292);
  --muted: oklch(0.95 0.018 292);
}
```

Tailwind reads the CSS variables and generates utilities from them. The color system uses OkLCH, which gives perceptually uniform colors — equal steps in lightness actually look equal, unlike hex or HSL.

Dark mode is just a class on `<html>`:

```css
.dark {
  --primary: oklch(0.75 0.2 292);
  --muted: oklch(0.16 0.026 292);
}
```

No configuration, no plugin, no `darkMode: 'class'` in a JS file. The CSS handles it.

## Deployment

The blog deploys to Vercel. The build command is `bun run build`, which runs `astro build` and outputs a static site to `dist/`. Vercel picks it up automatically.

```bash
bun run build    # outputs to dist/
bun run preview  # serves dist/ locally at http://localhost:4321
```

Zero configuration on the Vercel side — it detects Astro, runs the build, and deploys. The sitemap at `/sitemap-index.xml` is auto-generated by `@astrojs/sitemap`.
