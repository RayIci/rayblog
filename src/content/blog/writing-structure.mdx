Good structure is what separates a post you can scan from one you have to read start to finish to find what you need. This is a note on the patterns I've settled on — and a showcase of some of the components available in this blog.

## The problem with flat writing

Most first drafts come out flat — a sequence of paragraphs with no internal hierarchy. That's fine for a journal entry, but it doesn't serve someone arriving with a specific question.

Without headings, a reader has two choices: read everything, or leave. Headings create a third option: skim the structure, then dive into the section that matters.

<Callout type="tip" title="Start with an outline">
  Before writing any long post, I dump the H2 headings first. If I can't write
  five meaningful headings, the post isn't ready to write yet.
</Callout>

## What headings actually do

A good heading isn't a label. It's a promise. "How caching works" promises an explanation. "Caching" promises nothing.

The heading should be specific enough that someone skimming knows exactly what they're skipping — or exactly what they need to stop and read.

### The title (H1)

There's only one H1 and it's the post title. It should answer the question "what is this post about?" in under ten words.

### Section headings (H2)

H2s divide the post into major topics. I aim for 3–6 per post. If I have more, it usually means the post is trying to cover too much.

Each H2 should be able to stand alone as a meaningful statement. If removing the context of the post title makes the heading meaningless, it's probably too vague.

### Subsection headings (H3)

H3s go inside a section when there are distinct sub-cases or sub-topics worth separating. I use them sparingly — if every section has H3s, it's a sign the structure is over-engineered.

A good test: can a reader understand the H3 without reading the H2 above it? If yes, it probably belongs at H2 level. If no, it's a legitimate H3.

## Anchor links

Every heading on this blog has a permanent anchor link. Hover over any heading and a `#` will appear — click it to copy a direct link to that section.

This makes it easy to share a specific part of a long post, or to link to a section from elsewhere on the site.

## Math rendering

This blog supports LaTeX math via KaTeX. Inline formulas look like $E = mc^2$ and are rendered inline with the text. Block formulas are centered on their own line:

$$
\int_0^\infty e^{-x^2} dx = \frac{\sqrt{\pi}}{2}
$$

Inline: the quadratic formula is $x = \frac{-b \pm \sqrt{b^2 - 4ac}}{2a}$.

## Footnotes

Markdown supports footnotes[^1] natively. They're useful for asides that would break the flow of prose if included inline, or for citations[^2].

## Paragraph length

I write short paragraphs on purpose. Each paragraph should contain one idea. When I find myself writing a paragraph that contains a pivot ("but", "however", "on the other hand"), I split it.

Short paragraphs make the visual rhythm of the page easier to scan. They also force you to be more precise — you can't hide ambiguity in a 200-word paragraph.

## Lists vs. prose

I use lists for:

- Enumerable items where order matters
- Steps in a process
- Options or trade-offs being compared

I use prose for:

- Arguments that build on each other
- Explanations where the connection between ideas matters
- Anything where the flow of reading is part of the meaning

If I'm reaching for a list because I don't want to write transitions, that's a signal I need to think more carefully about how the ideas connect.

## Conclusion sections

I almost never write a "Conclusion" heading. If the post needs a summary, the post is probably too long. If the final section wraps things up naturally, adding a heading called "Conclusion" just announces that the post is ending — which the reader can already see.

The exception is long reference posts where a "Summary" or "TL;DR" at the end serves as a quick lookup for people who have already read the post once.

## What changes over time

These patterns aren't fixed. They've evolved from writing consistently and paying attention to what works. The main thing that's shifted is the direction: I used to default to more structure — more headings, more lists — and I've moved toward using structure only when it earns its place.

The goal is a post that reads cleanly even if you strip out every heading. The headings are there for navigation, not to hold the writing together.

[^1]: Footnotes appear at the bottom of the page with a back-link. Use them for tangents that would interrupt the main thread.
