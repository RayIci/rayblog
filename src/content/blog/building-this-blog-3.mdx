The first two parts covered the foundation stack and the reading experience. This final part covers everything that connects a post to the outside world: sharing, comments, embeds, and the automatic navigation features the layout handles.

## Share button

Every post footer has a share button. On mobile and desktop browsers that support the Web Share API, it opens the native OS share sheet:

```ts
await navigator.share({ title, url: window.location.href });
```

On browsers without Web Share API (most desktop), it falls back to copying the URL to the clipboard and briefly showing a "Copied!" confirmation.

Two additional links round out the share row: a pre-filled **X** post and a **LinkedIn** share link, both opening in a new tab.

## Edit on GitHub

Next to the share button is an "Edit on GitHub →" link that points directly to the MDX source file for the current post:

```ts
const slug = Astro.url.pathname.split("/").filter(Boolean).pop();
const editUrl = `https://github.com/RayIci/rayblog/blob/main/src/content/blog/${slug}.mdx`;
```

It's useful for fixing typos — click the link, edit the file on GitHub, open a PR.

## Giscus comments

Comments are powered by [Giscus](https://giscus.app), which stores discussions in GitHub Discussions on the repository. No external database, no moderation dashboard — just GitHub.

The component uses `next-themes` to match the blog's light/dark theme automatically:

```tsx title="src/components/GiscusComments.tsx"
import Giscus from "@giscus/react";
import { useTheme } from "next-themes";

export function GiscusComments() {
  const { resolvedTheme } = useTheme();
  return (
    <Giscus
      repo="RayIci/rayblog"
      mapping="pathname"
      reactionsEnabled="1"
      theme={resolvedTheme === "dark" ? "dark" : "light"}
      lang="en"
    />
  );
}
```

<Callout type="note">
  Giscus requires GitHub Discussions to be enabled on the repository. Visit
  giscus.app, enter the repo name, and copy the `repoId` and `categoryId` values
  into the component.
</Callout>

## GitHub repo cards

Any GitHub repository can be embedded as a card. It fetches metadata from the GitHub API at build time — stars, forks, description, primary language:

<GitHubCard repo="withastro/astro" />

<GitHubCard repo="giscus/giscus" />

The cards include no client JavaScript. All data is fetched during `astro build` and baked into the HTML.

## YouTube embeds

YouTube videos embed with a responsive `aspect-video` wrapper using the privacy-preserving `youtube-nocookie.com` domain, which doesn't set tracking cookies until the user clicks play:

<YouTube id="dQw4w9WgXcQ" title="Rick Astley — Never Gonna Give You Up" />

Usage in MDX is one line: `<YouTube id="dQw4w9WgXcQ" />`.

## Twitter/X embeds

Tweets embed using Twitter's oEmbed API, called at build time. The tweet HTML is baked into the static output — no Twitter JavaScript loads until the widget script runs:

<Tweet id="20" />

<Callout type="tip">
  If a tweet is deleted or the oEmbed API is unavailable during build, the
  component falls back to a plain link to the tweet URL.
</Callout>

## Prev / Next navigation

The post footer automatically shows links to the chronologically adjacent posts. No configuration — the layout fetches all posts, sorts by date, and finds the neighbors of the current post.

## Related posts

Below the Prev/Next block, up to three related posts appear based on tag overlap. Posts sharing at least one tag are considered related. This is also fully automatic — just tag your posts consistently.

## Series navigation

If a post belongs to a series (using `series` and `seriesOrder` in frontmatter), an index of all series posts appears at the top of the content, before the Table of Contents:

```yaml
series: "Building This Blog"
seriesOrder: 3
```

The current post is displayed as bold non-link text; the others are links. You can see this on all three parts of this series.
